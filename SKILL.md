# 盲审论文修改助手

## Overview

帮助研究生高效完成盲审论文的修改任务。本 Skill 定义了处理盲审论文修改的完整工作流程：从解析盲审意见、分析论文内容、生成修改建议，到输出规范的修改报告和标注 PDF。

**核心原则：先理解再修改，每条意见必须有回应，修改方案必须具体可执行。**

## When to Use

用户在以下场景应使用本 Skill：
- 用户提出"帮我修改盲审论文"、"帮我处理盲审意见"
- 用户提供盲审意见书和论文，要求逐条修改
- 用户需要生成规范的修改报告
- 用户收到盲审反馈后不知如何入手修改

**不要用于：**
- 没有盲审意见的论文润色（使用通用的写作润色 skill）
- 论文格式排版（使用专门的格式排版工具）

## Core Workflow

```
Step 1: 收集输入
  ├── 获取盲审版本论文（支持 PDF / Word）
  └── 获取盲审意见书（支持 PDF / 文本）

Step 2: 解析盲审意见
  ├── 逐条提取评审意见
  ├── 分类（创新性/实验设计/写作规范/格式/其他）
  └── 标注严重程度（必须修改/建议修改/可改可不改）

Step 3: 分析论文内容
  ├── 将评审意见映射到论文具体章节
  └── 定位问题所在位置

Step 4: 生成修改建议（调用 AI）
  ├── 针对每条意见生成具体的修改方案
  ├── 提供修改后的文本示例
  └── 标注修改等级

Step 5: 格式规范性检查
  ├── 检查论文字号、行距、页边距
  ├── 检查图表编号连续性
  ├── 检查参考文献格式
  └── 列出格式问题清单

Step 6: 输出报告
  ├── 生成 Markdown 格式的修改报告
  ├── 生成 HTML 可视化报告
  └── 生成标注版 PDF
```

## Input/Output Specifications

### 输入格式

| 文件类型 | 格式 | 说明 |
|---------|------|------|
| 论文 | PDF / DOCX | 盲审提交版本的论文 |
| 盲审意见 | PDF / TXT | 评审专家的盲审意见书 |

### 输出格式

| 输出 | 格式 | 说明 |
|------|------|------|
| 修改报告 | Markdown | 完整的逐条修改对照表 |
| 可视化报告 | HTML | 带样式的网页版报告 |
| 标注 PDF | PDF | 在原 PDF 上标注修改位置 |

## Review Comment Categorization

盲审意见分为以下五类，分类用于确定修改策略：

| 类别 | 判断关键词 | 修改策略 |
|------|-----------|---------|
| **创新性** | 创新、贡献、新颖、新意、研究价值 | 补充创新点论述，加强文献对比，突出差异化贡献 |
| **实验设计** | 实验、数据、方法、模型、样本、验证 | 补充实验细节，增加对比实验，完善数据分析 |
| **写作规范** | 写作、逻辑、结构、表达、语言 | 调整结构，优化逻辑，精简表达，统一术语 |
| **格式规范** | 格式、排版、图表、参考文献、字体 | 统一格式，规范引用，调整排版 |
| **其他** | 无法归入以上类别的意见 | 根据具体内容灵活处理 |

## Suggestion Generation Guidelines

生成修改建议时遵循以下原则：

1. **具体可操作**：不要说"需要改进创新性"，而要说"在第三章末尾增加一段论述，对比本文方法与现有方法的差异，突出三点创新贡献"
2. **有文本示例**：给出修改前后的文本对比
3. **尊重原文风格**：保持原论文的学术风格和表达习惯
4. **分级处理**：
   - **必须修改**：需要详细回应，给出完整的修改方案和示例
   - **建议修改**：给出主要修改方向，提供可选方案
   - **可改可不改**：简要说明理由，由用户决定

## Format Checking Rules

| 检查项 | 规范要求 | 检查方式 |
|--------|---------|---------|
| 摘要 | 200-300 字，包含目的、方法、结果、结论 | 文本长度统计 |
| 关键词 | 3-8 个，用分号或逗号分隔 | 正则提取+计数 |
| 章节结构 | 含绪论、文献综述、方法、实验、结论 | 目录结构分析 |
| 参考文献 | 格式统一，编号连续 | 正则匹配 |
| 图表编号 | 图/表编号连续无跳号 | 正则提取+排序检查 |

## Report Format

修改报告应包含以下四个部分：

### 一、盲审意见总览
```
共收到 N 条评审意见：
- 创新性: X 条
- 实验设计: X 条
- 写作规范: X 条
- 格式规范: X 条
- 其他: X 条
```

### 二、逐条修改建议
```
### 意见1
**评审意见**: [原文]
**分类**: 写作规范 | **严重程度**: 必须修改
**修改建议**:
[详细的修改方案和文本示例]
---
```

### 三、格式问题清单
```
| 问题类型 | 描述 | 位置 | 修改建议 |
```

### 四、修改总结
对整体修改情况进行概述，包括已处理意见数量、未处理意见及理由等。

## Quick Reference

```bash
# 自动解析盲审意见并生成修改报告

# 使用 Word 格式论文

```

## Common Mistakes

### ❌ 修改建议过于笼统
问题：说"建议改进创新性"但不说明如何改进
解决：必须给出具体的修改方向和文本示例

### ❌ 忽略部分评审意见
问题：只处理了容易修改的意见，回避了难处理的意见
解决：每条意见都必须回应，即使选择不修改也要说明理由

### ❌ 修改幅度过大
问题：完全重写原文，失去了原论文的特色
解决：只在与评审意见直接相关的部分做修改，保持原文的核心内容

### ❌ 格式与内容混为一谈
问题：在内容修改中夹杂格式问题，导致报告混乱
解决：内容修改在"逐条修改建议"中处理，格式问题单独在"格式问题清单"中列出

## Red Flags

- 用户只给了论文没给盲审意见 → 无法进行针对性修改
- 盲审意见模糊不清 → 需要先和用户确认具体意思
- 论文文件过大（>100页）→ 需要考虑分段处理
- 用户要求完全重写论文 → 超出本 skill 范围
- 多条意见相互矛盾 → 需要与用户沟通确定优先级

## The Bottom Line

**每条盲审意见都必须得到认真对待和具体回应。** 修改建议要具体到让用户可以直接按方案修改，而不只是给修改方向。这份 skill 的目标是让盲审修改变得系统化、可追踪、高质量。
