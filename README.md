# Automatic Revision Skill for Anonymous Paper Review

一个用于处理论文/学位论文盲审意见的 **Skill**。

把论文 DOCX 和盲审意见交给支持 Skill 的 Agent，它会根据本仓库的 `SKILL.md` 和配套流水线完成：

- 解析论文结构和盲审意见；
- 将每条意见定位到论文相关章节、图表或参考文献；
- 生成逐条修改方案、可写入正文的建议文本和盲审回应；
- 标出需要作者补充的真实实验、数据、图源或参考文献信息；
- 输出修改报告、盲审回应表、作者待补充事项和可审阅的修改方案。

这个项目不是论文代写工具。它不会编造实验结果、数据、参考文献、图表或页码。

## 快速开始

### 1. 下载项目

```bash
git clone git@github.com:dazzlingwuming/Automatic-Revision-Skill-for-Anonymous-Paper-Review.git
cd Automatic-Revision-Skill-for-Anonymous-Paper-Review
```

### 2. 安装依赖

```bash
pip install -r requirements.txt
```

### 3. 放入论文和盲审意见

例如：

```text
paper.docx
review.pdf
```

推荐使用 DOCX 作为论文主文件。盲审意见支持 PDF、DOCX、TXT 或 Markdown。

### 4. 让 Agent 使用这个 Skill

在支持 Skill 的 Agent 中直接说：

```text
请使用这个盲审论文修改 skill，处理 paper.docx 和 review.pdf，输出到 workdir/run1。
```

Agent 会读取 `SKILL.md`，并按项目内的工作流和脚本完成处理。你不需要手动理解每一个脚本。

## 输出内容

当前版本处理完成后，重点查看：

```text
workdir/run1/
├── outputs/
│   ├── 修改报告.md
│   ├── 盲审回应表.md
│   ├── 作者待补充事项.md
│   ├── 05_修改建议版.docx
│   └── 06_整合修改稿.docx
├── revision_plan_notes/
│   ├── R1-C001.md
│   ├── R1-C002.md
│   └── ...
└── audits/
    └── revision_solution_audit.json
```

说明：

- `修改报告.md`：总修改报告，包含每条意见的诊断、定位和修改方案。
- `盲审回应表.md`：可用于整理提交给评审/学院的回应说明。
- `作者待补充事项.md`：列出必须由作者提供真实材料的事项。
- `05_修改建议版.docx`：在论文副本中插入高亮修改建议，适合逐条审阅。
- `06_整合修改稿.docx`：把无需作者补材料的修改直接整合进论文副本，适合继续编辑。
- `revision_plan_notes/*.md`：每条盲审意见的可读修改卡片。

内部目录如 `paper/`、`assets/`、`artifacts/`、`revision_plans/`、`review/` 主要供流水线使用，通常不用直接阅读。

## 适用场景

适合：

- 硕士/博士论文盲审返修；
- 期刊或会议论文审稿意见返修；
- 需要生成逐条修改方案、回应表和 Word 建议稿的场景；
- 希望把“意见拆解、论文定位、修改建议、待补事项”标准化的场景。

不适合：

- 没有审稿/盲审意见的普通润色；
- 需要编造实验结果或参考文献信息的请求；
- 只提供 PDF 论文、没有 DOCX 原稿且要求高质量结构化修改的场景。

## 工作流程

这个 Skill 不是把论文和盲审意见简单丢给模型生成一段回答，而是把盲审返修拆成一套可追踪的流水线。

### 1. 输入检查

确认论文 DOCX、盲审意见文件和输出目录是否存在。论文优先使用 DOCX，因为 Word 文件能保留章节、段落、表格、图片和公式附近的结构信息。

### 2. DOCX-first 论文解析

系统会先解析论文原文，生成结构化材料：

- 论文全文 Markdown；
- 章节树；
- 每个章节的独立文本；
- 图表资产目录；
- 表格、图片和题注信息；
- 论文压缩摘要。

这样后续处理不是只依赖模型记忆整篇论文，而是能回到具体章节和原文证据。

### 3. 盲审意见解析

系统会把盲审意见拆成独立条目，并为每条意见标注：

- 意见编号；
- 评审来源；
- 原始意见文本；
- 问题类型，例如理论基础、方法设计、数据实验、格式规范、参考文献；
- 修改严重程度；
- 是否可能需要作者补充真实材料。

### 4. 意见与论文位置映射

每条盲审意见都会被映射到论文中的相关位置，包括：

- 核心修改章节；
- 支撑章节；
- 相关图表；
- 参考文献位置；
- 是否需要人工复核定位。

这一步解决的是“评审说的问题到底落在论文哪里”的问题。

### 5. 修改方案生成

系统会针对每条意见生成结构化修改方案，不只是给一句“建议补充说明”。方案会尽量包含：

- 问题诊断；
- 论文证据与原文定位；
- 修改策略；
- 可直接写入论文的新增或替换文本；
- 图表、实验、参考文献或格式修改方案；
- 同步修改位置，例如摘要、绪论、章节小结、总结；
- 给评审专家的回应话术；
- 需要作者补充的真实材料。

### 6. 质量审查

生成的修改方案会经过审查，重点检查：

- 是否真正回应了评审意见；
- 是否使用了论文原文证据；
- 是否只是泛泛建议；
- 是否缺少可执行修改内容；
- 是否把需要真实实验的数据写成了已完成结果；
- 是否对评审过度承诺。

如果方案太浅、只有口号式建议，应该继续打回重写。

### 7. 输出整理

最后输出给用户阅读和继续修改的材料：

- 总修改报告；
- 盲审回应表；
- 作者待补充事项；
- 每条意见的修改卡片；
- 当前版本中的 Word 修改建议版和整合修改稿。

## 相比直接在网页端对话的优势

当然，你也可以直接把论文和盲审意见发给网页端模型，让它帮你分析。但这个 Skill 的目标不是替代一次问答，而是把盲审返修变成一个可复核、可重复、可继续扩展的工作流。

### 1. 不靠一次性长对话记忆

网页端直接对话通常依赖模型在当前聊天里理解全文。论文一长，章节、图表、公式、参考文献和评审意见很容易混在一起。

这个 Skill 会先把 DOCX 拆成结构化文件，后续每条意见都能回到具体章节、图表和原文片段，减少“看起来说得对，但找不到原文依据”的问题。

### 2. 每条意见都有独立追踪

网页端问答很容易把多条盲审意见混合回答。这个 Skill 会把意见拆成独立编号，每条意见都有自己的诊断、定位、修改动作、回应话术和待补事项。

这样用户可以清楚看到：哪条意见已处理，哪条需要补实验，哪条只需要格式修改，哪条需要参考文献核实。

### 3. 输出不是一段回答，而是一组交付物

网页端通常给你一段文字，需要用户自己再复制、整理、拆表、做回应表。

这个 Skill 会直接整理成：

- `修改报告.md`
- `盲审回应表.md`
- `作者待补充事项.md`
- `revision_plan_notes/*.md`
- Word 副本

这些文件更适合后续和导师沟通、逐条检查、继续修改。

### 4. 更强调学术诚信边界

网页端如果提示不严，容易把“建议补实验”写成“已经完成实验”，或者补出看似真实的参考文献信息。

这个 Skill 明确把真实实验、数据、图源、参考文献核验放进 `作者待补充事项.md`，不把缺失材料伪装成已完成修改。

### 5. 可以沉淀为可复用流程

网页端对话通常是一次性的。换一篇论文、换一组盲审意见，就要重新组织提示词和上下文。

这个项目把解析、定位、生成、审查、输出都放进同一个 Skill 和脚本目录中。后续可以继续加入审查意见核查、多方案生成、按意见持续对话等能力，而不是每次从零开始问模型。

## 后续计划

这个项目后续会重点加强“审查、沟通、选择”能力，而不是只生成一个固定修改稿。

计划中的方向包括：

- **审查意见核查**：先判断盲审意见是否确实成立。系统会回到论文原文中查找证据，区分“论文确实缺失”“论文已有内容但表达不明显”“评审可能忽视了已有说明”“需要作者进一步确认”等情况。
- **多方案修改**：对于同一条盲审意见，不只给一个建议，而是生成多种可选解决方案。每种方案都要包含具体修改内容、适用场景、优缺点和预期效果，方便作者比较后自己决定。
- **修改前沟通**：在正式生成修改内容前，允许用户补充自己的判断、导师意见、实验限制或不想改动的边界，让修改方案更贴近真实论文情况。
- **方案后再斟酌**：模型给出多套方案后，用户可以继续追问、合并、否定或调整某个方案，再生成更精细的版本。
- **不默认生成整合 Word 稿**：未来交互式版本会优先输出“审查结论 + 多方案修改内容 + 具体正文效果预览”，不再默认生成 `05_修改建议版.docx` 和 `06_整合修改稿.docx`。是否写入 Word 由用户确认后再执行。
- **按意见持续对话**：每条盲审意见可以对应一个独立的修改上下文。用户如果对某一条修改不满意，可以让主程序重新进入对应问题的修改上下文，继续深入讨论、重写或扩展，而不是整篇论文重新跑一遍。

目标是把它从“一次性生成修改包”的工具，逐步变成一个可对话、可比较、可复核的盲审返修工作台。

## 高级用法：直接运行脚本

如果你想调试流水线，或在没有 Agent 的环境中先跑确定性流程，可以直接执行：

```bash
python scripts/run_pipeline.py --paper-docx ./paper.docx --review ./review.pdf --out workdir/run1 --title "论文标题" --mode full
```

常用模式：

```text
prepare  只做准备：解析 DOCX、解析意见、生成初始修改方案
report   基于已有 revision_plans 生成报告和 Word 输出
full     prepare + report
```

说明：`full` 可以生成稳定的结构化草稿和输出文件；如果要达到更高质量的逐条深度修改效果，应让 Agent 按 `SKILL.md` 中的 deep planner/auditor 流程继续深化每条意见。

## 学术诚信边界

本项目不会，也不应该：

- 编造实验结果；
- 编造数据；
- 编造参考文献信息；
- 编造图表源文件；
- 编造页码；
- 在未实际修改时声称“已修改”。

如果某条盲审意见需要真实实验、真实数据、导师决策、图源文件或参考文献核实，系统会把它列入 `作者待补充事项.md`。

## 项目结构

```text
.
├── SKILL.md          # Skill 入口
├── agent_specs/      # Agent 工作流和角色定义
├── scripts/          # 确定性命令行工具
├── src/              # Python 实现

```

`.claude/` 目录是兼容适配内容，不影响把本仓库作为通用 Skill 使用。

## 当前限制

- 推荐始终提供 DOCX；PDF-only 模式结构化质量较低。
- DOCX 本身没有可靠页码信息；如需精确页码，建议额外提供排版一致的 PDF 做对照。
- `05_修改建议版.docx` 是高亮建议版，不是 Word 修订模式 tracked changes。
- `06_整合修改稿.docx` 只会整合不需要作者补真实材料的修改。
